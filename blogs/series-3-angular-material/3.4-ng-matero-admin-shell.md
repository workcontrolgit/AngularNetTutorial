# Why We Didn't Build the Admin Shell from Scratch: ng-matero Explained

## How a Starter Template Provides the Responsive Layout, Theme Customizer, and Date Adapter — So You Can Focus on Features

Every Angular admin app needs the same boilerplate: a toolbar, a collapsible sidebar, a responsive layout that switches from side-nav on desktop to a drawer on mobile, and a theme toggle. Building all of that from scratch takes days and delivers nothing unique to your product.

The **TalentManagement** app skips that work entirely by scaffolding from [ng-matero](https://ng-matero.github.io/ng-matero/) — an open-source Angular admin template built on top of Angular Material. The result: a production-quality shell in `src/app/theme/`, ready on day one, while all development effort goes into the actual domain features.

This article walks through exactly what ng-matero contributes to the app — the `AdminLayout`, the `SettingsService`-driven options system, the `Customizer` panel, and the date-fns adapter — and where Angular Material ends and ng-matero begins.

## 🏗️ What Is ng-matero?

ng-matero is an Angular admin starter template and component library with two distinct parts:

**The scaffold (`ng-matero`):**
A pre-built admin shell — header, sidebar, sidemenu, breadcrumbs, theme customizer, auth layout. Scaffolded into `src/app/theme/` at project creation and owned by the app from that point on. You can modify every file.

**The extensions (`@ng-matero/extensions`):**
A library of UI components that Angular Material doesn't include — advanced select (`mtx-select`), drawer (`MtxDrawer`), progress, date-time picker, grid, and more. These are imported as packages and receive updates independently.

The relationship:

```
Angular CDK
  └── Angular Material
        └── ng-matero (theme scaffold + extensions)
```

ng-matero sits on top of Angular Material. Everything in the shell uses Material components internally (`mat-sidenav`, `mat-toolbar`, `mat-icon`). ng-matero adds the wiring, the layout logic, and the additional components that Material doesn't provide.

---

## 📦 Three Packages in Use

```json
"ng-matero": "^20.2.0",
"@ng-matero/extensions": "^20.3.0",
"@ng-matero/extensions-date-fns-adapter": "^20.0.0"
```

**`ng-matero`** — the starter template. After scaffolding, the source lives in `src/app/theme/`. The package itself is a dev-time tool; the generated files are what matters.

**`@ng-matero/extensions`** — the component library. Imported directly into components. The app uses two sub-packages:
* `@ng-matero/extensions/drawer` — the `MtxDrawer` service used by `Customizer`
* `@ng-matero/extensions/select` — `MtxSelectModule` inside the dormant Formly combobox type

**`@ng-matero/extensions-date-fns-adapter`** — replaces Angular Material's native `NativeDateAdapter` with a date-fns implementation for consistent date parsing and formatting.

## 🗂️ The Theme Directory

After scaffolding, all shell components live in `src/app/theme/`:

```
src/app/theme/
├── admin-layout/        ← root shell component
├── auth-layout/         ← login/callback layout (no sidebar)
├── customizer/          ← floating settings panel
├── header/              ← top toolbar
├── sidebar/             ← left nav panel + user panel
├── sidebar-notice/      ← right notice panel
├── sidemenu/            ← accordion nav menu
├── topmenu/             ← horizontal top nav (alternative to sidebar)
└── widgets/             ← branding, user button, translate button, notifications
```

The app has **two layouts**: `AdminLayout` (authenticated pages with sidebar) and `AuthLayout` (login/callback pages with no chrome). The router picks which layout to use via the route configuration.

## 🖥️ AdminLayout: The Shell Orchestrator

`AdminLayout` is the root component for all authenticated pages. Its job: assemble the three-panel shell and respond to screen size changes.

### The Template Structure

```html
<div class="matero-container-wrap"
  [class.matero-sidenav-collapsed]="options.sidenavCollapsed && options.navPos !== 'top'"
  [class.matero-navbar-side]="options.navPos === 'side'"
  [class.matero-navbar-top]="options.navPos === 'top'"
  [class.matero-header-above]="options.headerPos === 'above'"
  [class.matero-header-fixed]="options.headerPos === 'fixed'"
  [dir]="options.dir">

  <ng-progress ngProgressRouter />

  <!-- Header Above variant -->
  @if (options.showHeader && options.headerPos === 'above') {
    <app-header [showBranding]="true" ... />
  }

  <mat-sidenav-container class="matero-container" autosize autoFocus>
    <mat-sidenav #sidenav class="matero-sidenav"
      [mode]="isOver ? 'over' : 'side'"
      [opened]="options.navPos==='side' && options.sidenavOpened && !isOver">
      <app-sidebar ... />
    </mat-sidenav>

    <mat-sidenav #sidenavNotice position="end" mode="over">
      <app-sidebar-notice />
    </mat-sidenav>

    <mat-sidenav-content #content class="matero-content">
      @if (options.showHeader && options.headerPos !== 'above') {
        <app-header ... />
      }
      @if (options.navPos === 'top') {
        <app-topmenu />
      }
      <main class="matero-page-content">
        <router-outlet />
      </main>
    </mat-sidenav-content>
  </mat-sidenav-container>
</div>
```

**`[class.matero-*]` bindings** — every layout variation is expressed as a CSS class on the root `div`. The stylesheet does the heavy lifting; the TypeScript only toggles class names. This keeps visual logic in CSS where it belongs.

**`[dir]="options.dir"`** — a single binding on the root element flips the entire layout between LTR and RTL. Angular CDK's `BidiModule` propagates the direction to all child components automatically.

**`ng-progress ngProgressRouter`** — the `NgProgressRouter` directive from `ngx-progressbar` automatically shows a thin progress bar during route transitions. One line in the template handles all page navigation feedback.

**Two sidnavs** — the left sidenav (`#sidenav`) holds the navigation. A second right-side sidenav (`#sidenavNotice`) holds a notification panel. Both use Angular Material's `MatSidenav`.

### Responsive Breakpoints

```typescript
const MOBILE_MEDIAQUERY  = 'screen and (max-width: 599px)';
const TABLET_MEDIAQUERY  = 'screen and (min-width: 600px) and (max-width: 959px)';
const MONITOR_MEDIAQUERY = 'screen and (min-width: 960px)';

this.breakpointObserver
  .observe([MOBILE_MEDIAQUERY, TABLET_MEDIAQUERY, MONITOR_MEDIAQUERY])
  .subscribe(state => {
    this.options.sidenavOpened  = true;
    this.isMobileScreen         = state.breakpoints[MOBILE_MEDIAQUERY];
    this.options.sidenavCollapsed = state.breakpoints[TABLET_MEDIAQUERY];
    this.isContentWidthFixed    = state.breakpoints[MONITOR_MEDIAQUERY];
  });
```

`BreakpointObserver` from Angular CDK fires whenever the screen crosses a breakpoint. The subscription updates three booleans:

* **Mobile** (`≤599px`): `isMobileScreen = true` → sidenav switches to `mode="over"` (drawer that overlays content, closes after navigation)
* **Tablet** (`600–959px`): `sidenavCollapsed = true` → sidenav collapses to icon-only width
* **Monitor** (`≥960px`): `isContentWidthFixed = true` → content area uses fixed-width layout

No media queries in TypeScript. No `window.innerWidth` checks. `BreakpointObserver` handles all screen size detection.

### Auto-Close on Navigation

```typescript
this.router.events
  .pipe(filter(event => event instanceof NavigationEnd))
  .subscribe(() => {
    if (this.isOver) {
      this.sidenav().close();   // close drawer on mobile after navigation
    }
    this.content().scrollTo({ top: 0 });  // scroll content to top
  });
```

On mobile, the sidenav is in overlay mode. After the user taps a menu item and navigation completes, the drawer closes automatically. The content area also scrolls to top — preventing the new page from inheriting the previous page's scroll position.

## ⚙️ AppSettings: One Object Drives Everything

`SettingsService` holds an `AppSettings` object that controls every layout variation:

```typescript
interface AppSettings {
  navPos:          'side' | 'top';          // sidebar vs top menu
  dir:             'ltr' | 'rtl';           // text direction
  theme:           'light' | 'dark' | 'auto';
  showHeader:      boolean;
  headerPos:       'above' | 'fixed' | 'static';
  showUserPanel:   boolean;
  sidenavOpened:   boolean;
  sidenavCollapsed: boolean;
}
```

`AdminLayout` reads this object:

```typescript
options = this.settings.options;
```

Every `[class.matero-*]` binding in the template reads a property from `options`. Change a property → class toggles → CSS responds. The settings are persisted to `localStorage` so the user's preferences survive a page refresh.

## 🎨 Customizer: Live Theme Settings with MtxDrawer

The floating gear icon in the bottom-right corner opens a settings panel. This is the `Customizer` component — the one place in the app that uses `@ng-matero/extensions`.

### The Floating Button

```html
<div class="customizer-handle" cdkDrag (cdkDragStarted)="onDragStart($event)">
  <button matFab (click)="openPanel(customizerPanel)">
    <mat-icon>settings</mat-icon>
  </button>
</div>
```

`cdkDrag` (Angular CDK) makes the button draggable — users can reposition it anywhere on the screen. The `onDragStart` guard prevents a drag event from also triggering a click.

### Opening the Drawer

```typescript
private readonly drawer = inject(MtxDrawer);
private drawerRef?: MtxDrawerRef;

openPanel(templateRef: TemplateRef<any>) {
  if (this.dragging) { this.dragging = false; return; }

  this.drawerRef = this.drawer.open(templateRef, {
    position: this.form.get('dir')?.value === 'rtl' ? 'left' : 'right',
    width: '320px',
  });

  this.drawerRef.afterOpened().subscribe(() => {
    this.formSubscription = this.form.valueChanges.subscribe(value => {
      this.sendOptions(this.form.getRawValue());
    });
  });

  this.drawerRef.afterDismissed().subscribe(() => {
    this.formSubscription.unsubscribe();
  });
}
```

`MtxDrawer.open()` works like Angular Material's `MatDialog.open()` — it takes a `TemplateRef` and a config object, and returns a `MtxDrawerRef`. The `position` is direction-aware: RTL apps get the drawer on the left.

The reactive form subscription only runs while the drawer is open (`afterOpened` → `afterDismissed`). Every form value change emits immediately — no Save button. The layout updates in real time as the user changes options.

### The Settings Form

```html
<form [formGroup]="form">
  <mat-radio-group formControlName="theme">
    <mat-radio-button value="light">Light</mat-radio-button>
    <mat-radio-button value="dark">Dark</mat-radio-button>
    <mat-radio-button value="auto">Auto</mat-radio-button>
  </mat-radio-group>

  <mat-slide-toggle formControlName="showHeader">Visible</mat-slide-toggle>

  <mat-radio-group formControlName="headerPos">
    <mat-radio-button value="above">Above</mat-radio-button>
    <mat-radio-button value="fixed">Fixed</mat-radio-button>
    <mat-radio-button value="static">Static</mat-radio-button>
  </mat-radio-group>

  <mat-radio-group formControlName="navPos">
    <mat-radio-button value="side">Side</mat-radio-button>
    <mat-radio-button value="top">Top</mat-radio-button>
  </mat-radio-group>

  <mat-radio-group formControlName="dir">
    <mat-radio-button value="ltr">LTR</mat-radio-button>
    <mat-radio-button value="rtl">RTL</mat-radio-button>
  </mat-radio-group>
</form>
```

Standard Angular Material form controls bound to a reactive `FormGroup` initialized from `SettingsService.options`. The form is the single source of truth for the customizer — when the drawer closes, the form subscription unsubscribes and no further updates fire.

## 📅 The Date-fns Adapter

Angular Material's date picker needs a date adapter to know how to parse strings into dates and format dates for display. The default `NativeDateAdapter` uses JavaScript's `Date` constructor, which has inconsistent locale behavior across browsers.

`@ng-matero/extensions-date-fns-adapter` replaces it with [date-fns](https://date-fns.org/) — a modern, locale-aware date utility library.

```typescript
// app.config.ts
provideDateFnsDatetimeAdapter({
  parse: {
    dateInput: 'yyyy-MM-dd',
  },
  display: {
    dateInput:         'yyyy-MM-dd',
    monthYearLabel:    'yyyy MMM',
    dateA11yLabel:     'LL',
    monthYearA11yLabel: 'yyyy MMM',
  },
}),
```

**`parse.dateInput`** — the format date-fns uses to parse a string the user types into the date input field.

**`display.dateInput`** — the format used when displaying a selected date back in the input field.

**Why this matters for the API:** The TalentManagement API expects ISO date strings (`"1990-05-15"`). The adapter ensures every date the user picks is formatted as `yyyy-MM-dd` before the form value is sent to the API — no `Date` object serialization quirks.

Registering the adapter in `app.config.ts` applies it globally — every `mat-datepicker` in the application uses date-fns formatting automatically. No per-component configuration needed.

## 🔀 What ng-matero Provides vs. What You Build

**Provided by ng-matero (in `src/app/theme/`):**
* Responsive three-panel layout (header + sidenav + content)
* Breakpoint-aware sidebar behavior (side / collapsed / drawer)
* Live theme customizer panel with `MtxDrawer`
* Top menu alternative to sidebar
* User panel in sidebar
* Auth layout for login pages
* Navigation progress bar integration

**Built by the app (in `src/app/routes/`):**
* Employee, department, position, salary range CRUD
* Data tables with server-side pagination and filtering
* Reactive forms with validation
* Confirm dialog for delete actions
* Role-based UI with `*appHasRole`
* OIDC authentication and HTTP interceptor

The split is clean: ng-matero owns the shell, the app owns the domain. Neither bleeds into the other's territory.

## 🎯 Key Design Decisions

**Scaffold once, own forever** — ng-matero generates `src/app/theme/` and then stays out of the way. The generated files belong to the app. You can delete, rename, or rewrite any of them without worrying about package updates. ng-matero the package is just the scaffolding tool.

**CSS class bindings over inline styles** — `[class.matero-header-fixed]="options.headerPos === 'fixed'"` is a one-line binding. The stylesheet handles what "fixed header" actually means (position, z-index, box-shadow). Separating intent (the binding) from implementation (the CSS) keeps the TypeScript readable.

**`BreakpointObserver` over media queries in TypeScript** — responding to screen size changes via `BreakpointObserver` means the logic runs exactly when breakpoints cross, not on every resize event. Angular CDK handles the subscription lifecycle.

**`MtxDrawer` over `MatDialog` for settings panels** — dialogs are centered, modal, and interrupt workflow. A drawer slides in from the side, stays dismissible, and lets users see the effect of their changes in the background. The right component for the right UX pattern.

**Live form sync without a Save button** — subscribing to `form.valueChanges` inside `drawerRef.afterOpened()` means every radio button or toggle change is immediately reflected in the layout. Users see results instantly. The subscription is cleaned up in `afterDismissed()` so no stale observers remain.
