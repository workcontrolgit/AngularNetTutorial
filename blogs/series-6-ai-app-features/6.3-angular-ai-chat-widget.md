# Add an AI Chat Widget to Angular with Material Design

## How to Build a Two-Mode AI Assistant UI Using Angular Standalone Components

A great AI backend is worthless without a UI to access it. This article adds a polished chat interface to the TalentManagement Angular app — one tab for general-purpose questions, a second tab for live HR data insights backed by the endpoint we built in Article 6.2.

## 🎯 What We're Building

A new **AI Chat** page accessible from the sidebar with two tabs:

**Tab 1 — General Chat:**
* Text input → `POST /api/v1/ai/chat` → displays reply
* Conversation history accumulates per session
* Loading spinner while Ollama is inferring
* Error message if the API is unreachable or AI is disabled

**Tab 2 — HR Insights:**
* Four suggestion buttons with pre-filled HR questions
* Text input → `POST /api/v1/ai/hr-insight` → displays data-grounded answer
* Shows inference time (milliseconds) below each AI reply
* Same loading / error pattern as Tab 1

Both tabs are hidden behind `environment.aiEnabled = false` by default. Readers without Ollama see an info banner instead of broken UI.

## 🚀 Implementation

### Step 1: Add `aiEnabled` to the Environment

Add the feature flag to both environment files.

**`src/environments/environment.ts`:**

```typescript
export const environment = {
  production: false,
  apiUrl: 'https://localhost:44378/api/v1',
  identityServerUrl: 'https://localhost:44310',
  clientId: 'TalentManagement',
  scope: 'openid profile email roles app.api.talentmanagement.read app.api.talentmanagement.write',

  // Feature Flags
  allowAnonymousAccess: true,
  aiEnabled: false,   // ← set to true to activate AI features
};
```

**`src/environments/environment.prod.ts`:**

Add the same `aiEnabled: false` line under `// Feature Flags`. Production defaults to off.

**Why default to false?** Readers who are following the original Series 0–5 tutorial don't have Ollama running. If the chat widget made API calls with `AiEnabled: false` in the API, every request would return `503 Service Unavailable` — a broken experience. Defaulting the flag to `false` in the Angular environment means the chat UI is never shown to those readers, so their app continues to work exactly as it did before.

### Step 2: Create the AI Service

The AI endpoints don't fit the CRUD pattern in `BaseApiService<T>` — there's no entity to list, create, update, or delete. Create a focused service instead.

**`src/app/services/api/ai.service.ts`:**

```typescript
import { Injectable, inject } from '@angular/core';
import { HttpClient } from '@angular/common/http';
import { Observable } from 'rxjs';
import { environment } from '../../../environments/environment';

export interface AiChatResponse {
  reply: string;
}

export interface HrInsightDto {
  question: string;
  answer: string;
  executionTimeMs: number;
}

export interface HrInsightResult {
  succeeded: boolean;
  data: HrInsightDto;
  message?: string;
}

@Injectable({
  providedIn: 'root',
})
export class AiService {
  private http = inject(HttpClient);
  private apiUrl = environment.apiUrl;

  chat(message: string, systemPrompt?: string): Observable<AiChatResponse> {
    return this.http.post<AiChatResponse>(`${this.apiUrl}/ai/chat`, {
      message,
      systemPrompt,
    });
  }

  hrInsight(question: string): Observable<HrInsightResult> {
    return this.http.post<HrInsightResult>(`${this.apiUrl}/ai/hr-insight`, {
      question,
    });
  }
}
```

**Key design decisions:**

* **Does not extend `BaseApiService<T>`** — The AI endpoints are not CRUD endpoints. Inheriting the base service would pull in methods (`getAll`, `getById`, `create`, `update`, `delete`) that don't apply here and would be misleading to future developers.
* **`inject()` not constructor injection** — Consistent with the rest of this codebase. The `inject()` function works in the constructor phase and produces cleaner code with no boilerplate constructor.
* **`AiChatResponse` and `HrInsightResult` defined here** — These interfaces only exist to type the HTTP response. They live in the service file to keep the model close to the code that uses it.

**Add to the barrel export** in `src/app/services/api/index.ts`:

```typescript
export * from './ai.service';
export * from './base-api.service';
// ... existing exports
```

### Step 3: Create the Chat Component

Create the folder `src/app/routes/ai-chat/` with three files.

**`ai-chat.component.ts`:**

```typescript
import { Component, OnDestroy, inject } from '@angular/core';
import { CommonModule } from '@angular/common';
import { FormsModule } from '@angular/forms';
import { MatCardModule } from '@angular/material/card';
import { MatIconModule } from '@angular/material/icon';
import { MatButtonModule } from '@angular/material/button';
import { MatInputModule } from '@angular/material/input';
import { MatFormFieldModule } from '@angular/material/form-field';
import { MatProgressSpinnerModule } from '@angular/material/progress-spinner';
import { MatTabsModule } from '@angular/material/tabs';
import { MatDividerModule } from '@angular/material/divider';
import { Subject } from 'rxjs';
import { takeUntil } from 'rxjs/operators';
import { PageHeader } from '@shared';
import { AiService } from '../../services/api/ai.service';
import { environment } from '../../../environments/environment';

export interface ChatMessage {
  role: 'user' | 'assistant';
  content: string;
  executionTimeMs?: number;
}

@Component({
  selector: 'app-ai-chat',
  standalone: true,
  templateUrl: './ai-chat.component.html',
  styleUrl: './ai-chat.component.scss',
  imports: [
    CommonModule,
    FormsModule,
    MatCardModule,
    MatIconModule,
    MatButtonModule,
    MatInputModule,
    MatFormFieldModule,
    MatProgressSpinnerModule,
    MatTabsModule,
    MatDividerModule,
    PageHeader,
  ],
})
export class AiChatComponent implements OnDestroy {
  private aiService = inject(AiService);
  private destroy$ = new Subject<void>();

  aiEnabled = environment.aiEnabled;

  chatMessages: ChatMessage[] = [];
  chatInput = '';
  chatLoading = false;
  chatError = '';

  hrMessages: ChatMessage[] = [];
  hrInput = '';
  hrLoading = false;
  hrError = '';

  sendChat(): void {
    const message = this.chatInput.trim();
    if (!message || this.chatLoading) return;

    this.chatMessages.push({ role: 'user', content: message });
    this.chatInput = '';
    this.chatLoading = true;
    this.chatError = '';

    this.aiService
      .chat(message)
      .pipe(takeUntil(this.destroy$))
      .subscribe({
        next: response => {
          this.chatMessages.push({ role: 'assistant', content: response.reply });
          this.chatLoading = false;
        },
        error: err => {
          this.chatError = err?.error?.detail ?? 'Failed to get a response. Is the API running with AiEnabled: true?';
          this.chatLoading = false;
        },
      });
  }

  sendHrInsight(): void {
    const question = this.hrInput.trim();
    if (!question || this.hrLoading) return;

    this.hrMessages.push({ role: 'user', content: question });
    this.hrInput = '';
    this.hrLoading = true;
    this.hrError = '';

    this.aiService
      .hrInsight(question)
      .pipe(takeUntil(this.destroy$))
      .subscribe({
        next: result => {
          if (result.succeeded && result.data) {
            this.hrMessages.push({
              role: 'assistant',
              content: result.data.answer,
              executionTimeMs: result.data.executionTimeMs,
            });
          } else {
            this.hrError = result.message ?? 'Unexpected response from HR insights endpoint.';
          }
          this.hrLoading = false;
        },
        error: err => {
          this.hrError = err?.error?.detail ?? 'Failed to get HR insights. Is the API running with AiEnabled: true?';
          this.hrLoading = false;
        },
      });
  }

  onChatKeydown(event: KeyboardEvent): void {
    if (event.key === 'Enter' && !event.shiftKey) {
      event.preventDefault();
      this.sendChat();
    }
  }

  onHrKeydown(event: KeyboardEvent): void {
    if (event.key === 'Enter' && !event.shiftKey) {
      event.preventDefault();
      this.sendHrInsight();
    }
  }

  clearChat(): void { this.chatMessages = []; this.chatError = ''; }
  clearHr(): void { this.hrMessages = []; this.hrError = ''; }

  ngOnDestroy(): void {
    this.destroy$.next();
    this.destroy$.complete();
  }
}
```

**Key design decisions:**

* **`private destroy$ = new Subject<void>()`** — Every subscription uses `.pipe(takeUntil(this.destroy$))`. When `ngOnDestroy` fires, it completes the subject, which completes all active subscriptions. Without this, navigating away from the page while an Ollama request is in-flight would leave the component alive in memory, and the callback would fire on a destroyed view.

* **Separate state per tab** — `chatMessages`/`chatLoading`/`chatError` for Tab 1, `hrMessages`/`hrLoading`/`hrError` for Tab 2. Sharing state between tabs would mean clearing one tab's history when switching — a surprising user experience.

* **Guard on `!message || this.chatLoading`** — The send button is also `[disabled]` in the template, but the guard in the method is a defensive check. If JavaScript fires the method via keyboard shortcut while the button is disabled, it won't dispatch a second request.

* **`err?.error?.detail`** — The .NET API returns `ProblemDetails` on error, which has a `detail` field. This extracts the human-readable message for display. The `??` fallback handles network errors where `err.error` may be null.

### Step 4: Register the Route

In `src/app/app.routes.ts`, import the component and add the route:

```typescript
import { AiChatComponent } from './routes/ai-chat/ai-chat.component';

// Inside the AdminLayout children array:
{ path: 'ai-chat', component: AiChatComponent },
```

**No route guard needed.** The component itself checks `environment.aiEnabled` and shows a banner instead of the chat UI when the flag is false. Users without Ollama can navigate to `/ai-chat` and see a clear explanation rather than a broken page.

## 🌟 Why This Matters

The component patterns here — standalone imports, `inject()`, `takeUntil`, separate per-feature state — are the same patterns used throughout this codebase (see the Employee list, Dashboard, Profile pages). Adding AI to Angular doesn't require learning new framework concepts. It's the same Angular patterns applied to a new kind of data source: an LLM.

**Transferable skills:**

* **Service decomposition** — Not every service needs to extend a base class. Focused services with one responsibility are easier to test and reason about.
* **Feature flags in Angular** — `environment.aiEnabled` is the client-side equivalent of `FeatureManagement:AiEnabled` in the API. The same pattern works for any progressive feature rollout.
* **RxJS teardown** — The `Subject` + `takeUntil` pattern applies to any component that makes HTTP calls or subscribes to long-lived observables.
